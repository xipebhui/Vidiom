# Next Product Requirement: Real Model Storyboard Acceptance

更新时间：2026-07-10 CST

交接对象：架构师

## 产品目标

Vidiom 下一版必须完成真实模型 Storyboard 主链路的可重复验收与发布封口。最新提交已经实现 Storyboard 生成、审阅、图像关联和导出相关能力，README 也记录了一次真实 `.env` smoke 通过；但本轮产品任务独立复验时，真实配置 smoke 在语言 agent provider 调用阶段超过三分钟未完成并被中断。因此，下一步仍应优先收口模型接入验收，而不是切换到其他 LibTV 功能。

本需求的目标是让用户可以在真实 `gpt-5.5` 与 `gpt-image-2` 配置下，完整走通“agent 生成脚本和拍摄包 -> Storyboard 生成 -> 项目图像生成或复验 -> 导出交付”的产品链路，并在长耗时、失败或未完成时看到可信状态。

## 用户价值

用户需要的不是只在测试里成立的 Storyboard，而是在真实模型环境中可用、可等待、可失败可解释、可导出的创作链路。完成本需求后，用户应能：

- 用真实 `gpt-5.5` 生成短剧 agent 输出。
- 用真实 `gpt-5.5` 将完成项目拆成结构化 Storyboard shots。
- 看到每个 shot 的画面、动作、对白/声音、角色、场景、道具、时长和图像 prompt。
- 继续确认 `gpt-image-2` 项目图像能力没有退化。
- 在模型长时间等待、失败或配置问题时看到明确状态，而不是空白或假成功。
- 导出包含真实成功 Storyboard 的项目包，供后续分镜图、视频和合成能力继续使用。

## 功能范围

### 1. 真实端到端 smoke 复验

架构师需要把下一轮需求拆成可验证的真实模型验收闭环，而不是新增其他 LibTV 功能。验收链路应覆盖：

- 使用 `HM_BASE_URL`、`HM_LLM_APIKEY` 和 `gpt-5.5` 完成 agent 项目运行。
- 使用同一语言模型配置完成 Storyboard 生成。
- 使用 `HM_BASE_URL`、`HM_IMG_APIKEY` 和 `gpt-image-2` 完成项目图像生成或回归验证。
- 刷新后仍能查看 Storyboard。
- 导出包包含真实成功 Storyboard 的摘要、shots、项目内资产和图像关联信息。

### 2. 长耗时、失败和未完成状态验收

真实 provider 可能长时间等待或失败。产品必须能让用户区分：

- 未开始。
- 生成中。
- 真实成功。
- 真实失败。
- 用户中断或任务未完成。
- 已有旧成功结果但本次生成未成功。

失败和未完成状态不得显示为成功。不得生成占位 Storyboard、假 shot 或静默成功。不得自动改走其他模型或假数据；如果架构师认为需要引入调用超时策略，应在方案中标为“需用户确认”。

### 3. Storyboard 审阅与导出闭环复验

在真实 Storyboard 成功后，用户应能完成最小审阅闭环：

- 查看 Storyboard 状态。
- 查看 shot 数量和每个 shot 的核心字段。
- 查看每个 shot 的审阅状态和 prompt 准备度。
- 查看项目内角色、场景、道具资产摘要。
- 查看或建立已有项目图像资产与 shot 的关联。
- 在交付视图或导出语境中确认 Storyboard 是否进入交付包。

### 4. 文档与交接状态同步

下一轮需要把真实验收结果同步到迭代记录和相关产品/架构文档，避免“实现状态、测试状态、真实验收状态”彼此不一致。

文档应明确：

- 哪些能力已在自动化测试中通过。
- 哪些能力已在真实 `.env` smoke 中通过。
- 若真实 smoke 未通过，失败或超时发生在哪个产品阶段。
- README 通过记录与本轮复验超时之间的产品解释。
- 架构师下一轮交给开发的任务是否基于最新提交继续收口。

### 5. 现有真实模型能力回归

现有能力必须保持可用：

- Agent 运行继续使用 `gpt-5.5`。
- 项目图像生成继续使用 `gpt-image-2`。
- 缺少语言或图像模型配置时继续显示可见失败状态。
- 现有项目创建、运行、暂停、修订、Review、图像生成和导出流程不应退化。

## 关键用户流程

1. 用户创建项目，填写一句话和 Brief。
2. 用户运行 agent，等待真实 `gpt-5.5` 生成脚本和拍摄包。
3. 用户进入 Storyboard 视图，触发 Storyboard 生成。
4. 用户看到生成中或长耗时状态，并能判断任务仍未完成。
5. 生成成功后，用户查看结构化 shot 列表和资产摘要。
6. 用户检查每个 shot 的 prompt 准备度和审阅状态。
7. 用户验证或生成一个项目图像资产，并可理解其与 Storyboard 的关系。
8. 用户导出项目包，包内包含真实成功 Storyboard 结果。
9. 如果模型配置缺失、provider 失败或任务被中断，用户看到明确状态和可理解错误信息。

## 体验要求

- Storyboard 必须呈现为镜头生产结构，而不是长文本。
- 生成中、长时间等待、失败、旧结果和成功状态应清晰区分。
- Shot 列表需要便于扫描：顺序、状态、画面、角色、场景和 prompt 准备度应一眼可见。
- 错误信息应说明失败阶段或配置变量名，但不得暴露密钥值。
- 不得出现假生成结果、占位成功结果或静默成功。
- 真实 smoke 结果应能被下一轮产品任务直接读取和判断。
- 已有 `gpt-image-2` 项目图像能力不能被隐藏或破坏。

## 产品验收标准

### 真实模型链路验收

- 使用真实 `.env` 配置时，agent 项目运行可完成并产生结构化脚本和拍摄包。
- Storyboard 生成使用 `HM_BASE_URL`、`HM_LLM_APIKEY` 和 `gpt-5.5`。
- Storyboard 输入覆盖项目 seed、Brief、agent 输出和已有图像资产摘要。
- 成功结果包含多个结构化 shot，并覆盖顺序、剧情关联、角色、场景、道具、画面描述、动作/表演、对白/声音、建议时长、视觉要求和图像 prompt。
- Storyboard 结果可持久化，刷新后仍可查看。
- `gpt-image-2` 项目图像生成可在同一轮验收中确认未退化。

### 状态与失败验收

- 长时间等待时，用户不会看到假成功状态。
- 缺少 `HM_BASE_URL` 或 `HM_LLM_APIKEY` 时，Storyboard 进入可见失败状态。
- Provider 返回错误、非结构化结果或任务被中断时，Storyboard 进入可理解的失败或未完成状态。
- 已有成功 Storyboard 后再次生成失败，不得让用户误以为旧结果是本次成功结果。
- 错误信息不得包含密钥值。

### Studio 审阅验收

- 用户可从 Studio 进入 Storyboard 视图或等价工作区。
- 用户可查看 Storyboard 状态、shot 列表、资产摘要和图像关联位置。
- 每个 shot 能展示 prompt 准备度和审阅状态。
- 旧项目没有 Storyboard 时，不得在界面或导出包中伪造 Storyboard。

### 导出验收

- 成功 Storyboard 会进入导出包。
- 导出包包含 shots、资产摘要和图像关联摘要。
- 未生成、失败或未完成的 Storyboard 不应被导出为成功产物。
- 若存在旧成功结果但本次失败，导出语义必须让用户知道当前结果来源。

### 回归验收

- 自动化检查继续通过。
- 现有 agent、项目图像生成、项目创建、运行、暂停、修订、Review 编辑、交付导出保持可用。
- README 或等价迭代记录明确写入真实 smoke 是否可重复通过；若复验未通过，写明发生阶段。

## 非目标

本轮不切换到其他 LibTV 对齐功能：

- 不做自由无限画布节点创建。
- 不做完整 shot 深度编辑器作为唯一目标。
- 不做跨项目资产库。
- 不做批量分镜图生成。
- 不做图像编辑、全景、多角度、打光、九宫格、标注或分镜组完整工具套件。
- 不做批量视频生成。
- 不做视频剪辑或视频合成。
- 不做音频生成、提取、变速或音视频分离。
- 不做导演台 3D 构图。
- 不做主体库、风格库、自定义风格、运镜控制或真人素材合规校验。
- 不做自动改用其他模型、假数据或占位结果。

## 与 LibTV 的对应关系

本需求对应 LibTV 脚本/故事板链路中的“真实模型拆解剧本为结构化分镜并形成可信上游”阶段：

- 从剧本或故事想法拆解为结构化 shot。
- 提取角色、场景、道具等项目资产摘要。
- 生成用于后续分镜图和视频片段的 prompt。
- 为批量分镜图、批量视频和视频合成建立可信上游。

LibTV 的导演台、视频合成、真人素材合规、图像编辑和资产库能力仍是后续差距，但在 Vidiom 真实模型 Storyboard 主链路通过验收前，不应成为下一轮主需求。
