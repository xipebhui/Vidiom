# Next Product Requirement: Real Model Storyboard Generation

更新时间：2026-07-10 CST

交接对象：架构师

## 产品目标

Vidiom 下一版必须把 Storyboard 从“已具备数据底座”推进到“用户可触发、可观察、可验收的真实模型生成能力”。完成项目应能使用 OpenAI-compatible `gpt-5.5`，基于真实 agent 输出生成结构化故事板 shot 列表，并在 Studio 和导出包中形成清晰闭环。

该需求的产品重点是继续完成真实模型接入验收，而不是切换到其他 LibTV 对齐功能。`gpt-image-2` 项目图像生成需保持可用，并作为 Storyboard 后续分镜图能力的回归基础；本轮不要求新增批量图像生成。

## 用户价值

用户已经可以用真实模型生成短剧脚本和首张项目图像，但还不能把结果推进到逐镜头生产。真实模型 Storyboard 生成可以让用户：

- 从完整脚本快速得到可审阅的 shot 列表。
- 看到每个 shot 的画面、动作、对白/声音、角色、场景、道具、时长和图像 prompt。
- 判断哪些镜头已经可以进入后续分镜图生成。
- 在模型失败或配置缺失时看到明确失败状态，而不是空白或假成功。
- 把真实模型生成结果、人工审阅和导出交付连接成可验收工作流。

## 功能范围

### 1. 真实模型 Storyboard 生成

完成项目应能触发 Storyboard 生成。生成必须使用：

- `HM_BASE_URL`
- `HM_LLM_APIKEY`
- `gpt-5.5`

生成输入应来自当前项目真实上下文，包括一句话、创作 Brief、Premise、Character、Beat、Script、Production 和已有项目图像资产摘要。生成结果应是结构化 Storyboard，而不是脚本文本简单拆段。

每个 shot 至少应表达：

- 镜头顺序。
- 对应剧情/节拍。
- 出场角色。
- 场景/地点。
- 关键道具。
- 画面描述。
- 动作或表演重点。
- 对白或声音摘要。
- 建议时长。
- 画幅/视觉风格要求。
- 用于图像生成的 prompt 草稿。
- prompt 是否已准备好进入后续图像生成。

### 2. 可见生成状态与错误状态

用户应能区分 Storyboard 的未生成、生成中、生成成功、生成失败和已有结果状态。

失败状态必须可理解，并绑定到 Storyboard 语境。缺少 `HM_BASE_URL`、`HM_LLM_APIKEY` 或 provider 返回错误时，不得生成假 Storyboard，不得展示空白成功态，不得静默跳过。

如果项目已有成功 Storyboard，再次生成失败时，产品语义上应避免让用户误以为旧结果是本次成功结果；具体状态呈现交给架构师判断。

### 3. Storyboard 审阅入口

Studio 应提供用户可进入的 Storyboard 视图或等价工作区。用户应能查看：

- Storyboard 当前状态。
- shot 数量和每个 shot 的核心字段。
- 每个 shot 的审阅状态和 prompt 准备度。
- 项目内角色、场景、道具资产摘要。
- shot 与已有图像资产的关联位置。

本轮要求 Storyboard 生成后可被真实审阅。shot 深度编辑、资产编辑和批量分镜图生成可由架构师按风险拆分，但产品验收至少要覆盖用户能看懂、检查和导出生成结果。

### 4. 与现有真实模型能力的回归关系

现有真实模型能力必须保持可用：

- Agent 运行继续使用 `gpt-5.5`。
- 项目图像生成继续使用 `gpt-image-2`。
- 缺少语言或图像模型配置时继续显示可见失败状态。
- 现有项目创建、运行、暂停、修订、Review、图像生成和导出流程不应退化。

### 5. 交付与导出

成功生成的 Storyboard 应在项目刷新后仍可查看，并进入项目交付语境。导出包应包含 Storyboard 摘要、shots、项目内资产摘要和图像关联摘要，便于用户或后续流程继续处理。

## 关键用户流程

1. 用户创建项目，填写一句话和 Brief。
2. 用户运行 agent，得到真实 `gpt-5.5` 生成的脚本和拍摄包。
3. 用户进入 Storyboard 视图，触发 Storyboard 生成。
4. 用户看到生成中状态。
5. 生成成功后，用户查看结构化 shot 列表和资产摘要。
6. 用户检查每个 shot 的 prompt 准备度和后续分镜图生成条件。
7. 用户导出项目包，包内包含真实模型生成的 Storyboard 结果。
8. 如果模型配置缺失或 provider 失败，用户看到明确失败状态和可理解错误信息。

## 体验要求

- Storyboard 必须呈现为镜头生产结构，而不是长文本。
- 生成状态、成功状态、失败状态和旧结果状态应清晰区分。
- Shot 列表需要便于扫描：顺序、状态、画面、角色、场景和 prompt 准备度应一眼可见。
- 错误信息应说明失败原因所在的配置或模型调用阶段，但不得暴露密钥值。
- 不得出现假生成结果、占位成功结果或静默成功。
- 已有 `gpt-image-2` 项目图像能力不能被隐藏或破坏。

## 产品验收标准

### 真实模型生成验收

- 完成项目可触发 Storyboard 生成。
- Storyboard 生成使用 `HM_BASE_URL`、`HM_LLM_APIKEY` 和 `gpt-5.5`。
- 生成输入覆盖项目 seed、Brief、agent 输出和已有图像资产摘要。
- 成功结果包含多个结构化 shot，并覆盖顺序、剧情关联、角色、场景、道具、画面描述、动作/表演、对白/声音、建议时长、视觉要求和图像 prompt。
- 生成结果可持久化，刷新后仍可查看。

### 失败状态验收

- 缺少 `HM_BASE_URL` 或 `HM_LLM_APIKEY` 时，Storyboard 进入可见失败状态。
- Provider 返回错误或非结构化结果时，Storyboard 进入可见失败状态。
- 失败时不得生成占位 shot、假成功 Storyboard 或静默成功。
- 错误信息不得包含密钥值。

### Studio 审阅验收

- 用户可从 Studio 进入 Storyboard 视图或等价工作区。
- 用户可查看 Storyboard 状态、shot 列表、资产摘要和图像关联位置。
- 每个 shot 能展示 prompt 准备度。
- 旧项目没有 Storyboard 时，不得在界面或导出包中伪造 Storyboard。

### 导出验收

- 成功 Storyboard 会进入导出包。
- 导出包包含 shots、资产摘要和图像关联摘要。
- 未生成或失败的 Storyboard 不应被导出为成功产物。

### 回归验收

- 现有 agent 运行仍使用 `gpt-5.5`。
- 现有项目图像生成仍使用 `gpt-image-2`。
- 现有项目创建、运行、暂停、修订、Review 编辑、项目图像生成、交付导出和自动化测试保持通过。
- 如本机 `.env` 可用，应完成一次真实配置 smoke 验证：agent 运行、Storyboard 生成、项目图像生成和导出链路均可被检查。

## 非目标

本轮不切换到其他 LibTV 对齐功能：

- 不做自由无限画布节点创建。
- 不做完整 shot 深度编辑器作为唯一目标。
- 不做跨项目资产库。
- 不做批量分镜图生成。
- 不做图像编辑、全景、多角度、打光、九宫格、标注或分镜组完整工具套件。
- 不做批量视频生成。
- 不做视频剪辑或视频合成。
- 不做音频生成、提取、变速或音视频分离。
- 不做导演台 3D 构图。
- 不做主体库、风格库、自定义风格、运镜控制或真人素材合规校验。

## 与 LibTV 的对应关系

本需求对应 LibTV 脚本/故事板链路中的“真实模型拆解剧本为结构化分镜”阶段：

- 从剧本或故事想法拆解为结构化 shot。
- 提取角色、场景、道具等项目资产摘要。
- 生成用于后续分镜图和视频片段的 prompt。
- 为批量分镜图、批量视频和视频合成建立可信上游。

本轮不追求 LibTV 的完整图像、视频、音频、导演台和自由画布能力；先把 Vidiom 当前 Storyboard 主链路接到真实 `gpt-5.5` 并完成用户可验收闭环。
